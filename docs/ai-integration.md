# AI integration

Two distinct surfaces are rendered in the same panel ([components/AIAssistantPanel.tsx](../components/AIAssistantPanel.tsx)) but only one is actually wired to an LLM. Don't confuse them.

## 1. Suggestion cards (top of panel) — MOCK

The two cards at the top ("Add fillets to sharp edges", "Thicken the side walls") are **hardcoded** in `useAppStore.initialSuggestions` ([store/useAppStore.ts](../store/useAppStore.ts)). They have hardcoded `operations[]` arrays with feature IDs, descriptions, and parameter values.

Clicking **Preview** flips the suggestion status to `'previewing'` (mutually exclusive — previewing one drops any other back to pending). Clicking **Accept** or **Reject** updates the status accordingly. No LLM call.

For a real product, these cards would be generated by a fine-tuned model that reads the part's geometry + DFM analysis and proposes operations in the same shape.

## 2. Chat (bottom of panel) — LIVE

Free-form chat at the bottom. POST to `/api/ai/chat` with `{ messages: ChatMessage[] }` where each message is `{ id, role: 'user' | 'assistant', content }`. The server route ([app/api/ai/chat/route.ts](../app/api/ai/chat/route.ts)):

- Prepends a static system prompt (hardcoded — describes the part as an L-bracket with snap-fit hook, draft, thin wall, etc.)
- Calls `https://api.openai.com/v1/chat/completions` with `model: 'gpt-4o-mini'`, `temperature: 0.7`, `max_tokens: 280`
- Awaits the full response (no streaming)
- Returns `{ reply: string }`

Frontend appends both the user message and the assistant reply to `useAppStore.chatMessages`. Stable IDs are auto-generated by the store (`msg-${Date.now()}-${counter}`) so React keys are stable across re-renders.

### Quick prompts
Three pre-baked prompts ("Add fillet", "Check draft", "Optimize") that submit a longer query when clicked. Just convenience; same chat endpoint.

### Env var
`OPENAI_API_KEY` in `.env.local`. The route reads `process.env.OPENAI_API_KEY` directly — it's never exposed to the client (note the lack of `NEXT_PUBLIC_` prefix).

If the key is missing or invalid, the route returns an error JSON; the frontend renders `Couldn't reach the model: <error>` as an assistant message rather than crashing.

## Why it's structured this way

The suggestion cards are *deterministic UI actions* — they need stable, reviewable operation lists tied to specific feature IDs. An LLM is a poor fit for that today (would hallucinate IDs). The chat is *open-ended Q&A* — LLM strength. Splitting the panel into "deterministic suggestions" + "Q&A chat" is the same pattern Copilot uses: actions are model-grounded but the chat is a fallback.

## Future work (not implemented)

- **Stream the chat response** — `gpt-4o-mini` supports streaming; we just don't use it. Low effort.
- **Feed the current analysis into the system prompt** — currently static. Should include the user's part name, current DFM score, list of unfixed issues, and any applied fixes. Then the AI can reference real numbers ("you're at 56/100; fixing the undercut would bring you to 82").
- **Generate suggestion cards from the LLM** — biggest jump. Would need a constrained-output format (JSON mode + a schema) so the model emits validated `Operation[]` arrays the workspace can actually apply.
- **Conversation memory across reloads** — `chatMessages` is in-memory only; reloads clear it.
- **Cost / token tracking** — no instrumentation today.
- **Anthropic API as an alternative** — the project's pre-commit hook scans for both OpenAI and Anthropic key formats; we could add Claude as a second model option.
